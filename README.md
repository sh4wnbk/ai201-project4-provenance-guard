# Provenance Guard

A backend service that estimates whether submitted text was written by a human or
generated by AI, scores how confident that estimate is, returns a plain-language
transparency label, and lets creators appeal a result. Built for AI 201 Project 4.

The detector is deliberately conservative: on a writing platform, falsely
labeling a real person's work as AI is worse than missing a bot, so the system is
built to refuse to accuse when its two signals don't strongly agree.

## Architecture

A submission takes this path:

```
POST /submit (text, creator_id)
  -> length guard            reject input too short to score
  -> stylometric signal      structural score, 0..1 (pure Python)
  -> short-circuit?          if clearly human, skip the LLM call
  -> LLM signal (Cerebras)   semantic score, 0..1
  -> scoring                 combined estimate + confidence
  -> corroboration guard     refuse to accuse on weak LLM backing
  -> label                   one of three plain-language variants
  -> audit log + status      append-only log; status set to "classified"
  -> response                content_id, attribution, confidence, label
```

An appeal takes a separate path:

```
POST /appeal (content_id, creator_reasoning)
  -> status store    set status to "under_review"
  -> audit log       append the appeal next to the original entry (original unchanged)
  -> response        confirmation
```

The two stores serve different purposes. The audit log is append-only and exists
to reconstruct what happened — an appeal adds a row, it never edits the original
classification. The status store is mutable and answers "what needs a human now,"
so a reviewer reads the appeal queue from it without replaying the whole log.

Code is split so each piece is testable on its own: `app.py` holds routes only,
`stylometric.py` and `scoring.py` and `labels.py` are pure functions with no
network, `llm_signal.py` takes the API client as a parameter so it can be mocked,
and `store.py` owns both stores. Tunable constants live in `config.py`.

## Detection Signals

The system uses two signals that measure different things.

**Stylometric (structural).** Measures *burstiness* — the coefficient of variation
of sentence length (standard deviation over mean of per-sentence word counts).
Human writing swings between short and long sentences; AI writing tends to hold a
steadier medium length. Low variation maps to a high AI score through a logistic
function. It runs in pure Python with no network call, so it's cheap, fast, and
immune to prompt injection — you can't inject instructions into a variance
calculation. What it misses: it detects "uniform and formal," not "AI." Formal,
technical, or non-native English writing has low sentence-length variation and
scores high-AI even when a person wrote it.

**LLM (semantic).** Cerebras. The model reads the text and returns an AI-likelihood
score from 0 to 1 in a fixed JSON schema. It reasons about meaning and coherence
rather than surface statistics, so it can sometimes recognize that a formal piece
is genuine human reasoning. What it misses: it's a black box with no auditable
reason, it can be confidently wrong, and — as the validation below shows — it
shares some of the stylometric signal's bias toward calling formal writing AI.

I dropped two metrics the course material lists as standard stylometric signals.
I computed type-token ratio (MATTR) on the four rubric test inputs before
committing to it; the scores clustered between 0.92 and 0.98 across all four —
the texts are too short for vocabulary-diversity to separate anything. Punctuation
density was fragile (it tracked register, not authorship). Burstiness was the only
structural metric that discriminated, so it's the backbone. Dropping MATTR against
the lecture's default was a measurement decision, not an oversight.

## Confidence Scoring

Two numbers come out of scoring, not one:

```
combined   = (sty + llm) / 2                              # AI-likelihood, sets direction
confidence = (1 - |sty - llm|) * (2 * |combined - 0.5|)   # how sure the verdict is
```

Confidence is high only when the signals **agree** (the `1 - |sty - llm|` term) and
the verdict is **strong** (far from 0.5). Either one failing pulls it down. This
keeps confidence separate from AI-likelihood: a confident "human" and a confident
"AI" are both high-confidence, and a genuinely ambiguous case is low-confidence in
either direction. Averaging for the direction is safe because disagreement is kept
in the confidence term — the case where one signal says 0.9 and the other 0.1
lands at confidence 0 and is reported Uncertain, not mislabeled.

The label is decided in this order:

1. LLM unavailable (failure or unparseable) -> Uncertain (fail-safe; never default to human)
2. Stylometric accuses (sty > 0.5) but LLM doesn't strongly back it (llm < 0.70) -> Uncertain
3. confidence >= 0.35 -> Likely AI or Likely human, by direction
4. otherwise -> Uncertain

`T_HIGH = 0.35` and `LLM_CORROBORATE_MIN = 0.70` are in `config.py`.

### Worked examples

Two submissions at different confidence levels, with the actual signal scores:

**High confidence — clearly human**

| field      | value        |
|------------|--------------|
| sty_score  | 0.126        |
| llm_score  | 0.150        |
| combined   | 0.138        |
| confidence | 0.707        |
| label      | Likely human |

Both signals agree (gap of 0.024) and the verdict is far from 0.5 (combined 0.138
is 0.362 below the midpoint), so confidence lands high at 0.707.

**Low confidence — formal human (the fairness case)**

| field        | value               |
|--------------|---------------------|
| sty_score    | 0.911               |
| llm_score    | 0.600 (measured)    |
| combined     | 0.756               |
| confidence   | 0.352               |
| label        | Uncertain           |
| audit_reason | weak_corroboration  |

The stylometric signal accuses (0.911) but the LLM only mildly agrees (0.600).
The corroboration guard fires — when sty > 0.5 and llm < 0.70, the verdict is
forced to Uncertain regardless of what the confidence gate would say. Without the
guard, confidence 0.352 just clears T_HIGH = 0.35 and the result would be Likely
AI, a false label by a margin of 0.002. This is the low-confidence half of the
pair and the system's primary known weakness — formal technical writing scores high
on both signals for structural reasons unrelated to authorship.

**How I validated the scores are meaningful.** The scoring tests run the four
rubric inputs through the pipeline and assert each lands in the right category for
the right reason. The stylometric half is pinned: burstiness has to reproduce
0.379 / 0.611 / 0.256 / 0.382 on the four inputs, and the AI scores 0.700 / 0.126
/ 0.911 / 0.693, or the tests fail — so a tokenization mistake can't silently skew
everything downstream. Two LLM scores are measured directly against Cerebras at
temperature 0: formal-human returns 0.600 and clear-AI returns 0.780, both stable
across five runs. The other two inputs (clear-human 0.15, lightly-edited-AI 0.55)
use estimates; both short-circuit or fall into the guard path before the score
matters, so precision there doesn't affect the label outcome. The corroboration
threshold (0.70) sits 0.080 above the formal-human score and 0.080 below the
clear-AI score — anchored on two measured points, not tuned to one. Separate tests
assert the guard fires (sty 0.8, llm 0.65 -> Uncertain), doesn't fire when the
LLM strongly backs the accusation (sty 0.8, llm 0.75 -> Likely AI), and doesn't
apply in the human direction.

## Transparency Label

The label describes what the signals found rather than asserting a verdict, and it
never shows the raw confidence number. The three variants:

**Likely AI**
> This text shows patterns we often see in AI-generated writing — very even
> sentence rhythm and uniform phrasing. This is an automated signal, not proof.
> If you wrote it yourself, you can ask us to review.

**Likely human**
> This text shows the natural variation we usually see in human writing. We didn't
> find signs of AI generation.

**Uncertain**
> We couldn't reach a confident read on this one. The signals disagreed — that
> often happens with formal, technical, or non-native English writing, which can
> resemble AI patterns without being AI. Nothing has been recorded against you.
> You can add context or request a review.

The AI variant is hedged harder than the human variant and is the one that offers
the appeal path, because a false AI label harms a real person while a false human
label only misses a bot. The uncertain variant names the writing it tends to
misread and says outright that nothing was logged against the creator, so an
appeal reads as adding context rather than fighting an accusation.

## Rate Limiting

`POST /submit` is limited to **10 per minute and 100 per day per IP**
(`Flask-Limiter`, in-memory storage).

The reasoning is tied to how a writing platform is actually used. A creator
submitting their own work does it a handful of times in a session, and even while
iterating — resubmit after an edit, check a revision — rarely exceeds a few per
minute. Ten per minute absorbs that legitimate bursting while a script hammering
the endpoint hits the wall immediately. One hundred per day bounds sustained abuse
from a single source without coming anywhere near what a real creator would do in
a day. In-memory storage is fine for a single-process demo; a production
deployment with multiple workers would move the limiter to Redis so the counts
hold across processes.

## Known Limitations

**Formal, technical, and non-native English writing.** This is the system's
weakest case, and it's measured. Both signals lean AI on this register: burstiness
scores it high because formal prose has uniform sentence length (the monetary-policy
test sample scored 0.911, the most AI-looking of all four inputs, and it's human),
and the Cerebras model also leans AI on it (0.600 measured), because formal writing
resembles its internal notion of "generated" text. The two signals are correlated
here rather than independent, so the disagreement that protects other ambiguous
cases doesn't appear. The corroboration guard catches it — stylometric accuses, the
LLM doesn't strongly back it, so the result routes to Uncertain instead of Likely
AI — but the appeal flow exists precisely because this population is where the
detector is least reliable.

**Short and poetic text.** Burstiness is a variation measure over sentence lengths,
so with only a handful of sentences one sentence swings the score hard and the
structural signal gets noisy. A short poem with deliberately even line lengths can
read as AI to the stylometric signal for a reason that has nothing to do with how
it was written. Input below the length guard is rejected; input just above it is
scored but unreliable, and the system leans on the LLM there.

## Spec Reflection

**Where the spec helped.** Writing the full plan before any code, with the
stylometric numbers pinned, gave every milestone a concrete gate. Burstiness had
to reproduce four specific values before it was wired into anything, so an error
in tokenization would have failed a test immediately instead of quietly skewing
every score built on top of it.

**Where the build diverged from the plan.** The plan assumed the LLM would score
formal-human text around 0.35, disagreeing with the stylometric 0.911 and routing
it to Uncertain. The pre-implementation validation (Gate 0) measured the real
Cerebras score at 0.600 — the signals were correlated, not independent, and the
formal-human case fell through to a false "Likely AI" by a margin of 0.002. The
original disagreement mechanism only annotated the audit log; it didn't gate the
verdict, so it had no power to stop this. I added an asymmetric corroboration guard
that wasn't in the original design: when the stylometric signal accuses but the LLM
doesn't strongly corroborate, the verdict is Uncertain. The estimate being wrong
was the most useful thing the planning produced, because the gate caught it before
a single line of LLM code was written.

## AI Usage

**Directing the design and overriding a fix (conversational assistant).** I worked
out the two-signal scoring, the confidence formula, and the label logic in
conversation, then handed sectioned spec briefs to Claude Code to implement. When
the Gate 0 measurement broke the design, Claude Code's first proposed fix was to
raise `T_HIGH` from 0.35 to 0.37 — a single-constant nudge that cleared the
formal-human case by 0.018 while the clear-AI margin shrank underneath it. I
overrode it. Tuning one threshold to one data point is overfitting, and the actual
bug was structural: the disagreement mechanism annotated the log instead of gating
the verdict. I directed the asymmetric corroboration guard instead, anchored on two
measured LLM scores rather than calibrated to rescue a single input.

**Dropping a signal against the course default.** I directed the assistant to
compute the candidate stylometric metrics on the four rubric inputs before
committing to any of them. The output showed MATTR clustered between 0.92 and 0.98
across all four — too short for it to discriminate. The course material lists TTR /
MATTR as a standard structural signal; I dropped it on the measurement and used
burstiness as the backbone instead of carrying a metric that wouldn't separate the
cases.

**Auditing a rewrite output before accepting it.** When `store.py` was converted
from in-memory dicts to SQLite, I explicitly checked that both stores survived —
not just the audit log, which M3 exercises immediately, but also the mutable status
store, which sits idle until M5. The idle store is exactly where a code-generation
tool tends to quietly drop things, because nothing in the current tests would catch
the omission. I read the generated code and confirmed `set_status`, `get_status`,
and `known_content_id` all hit a separate `status_store` table rather than being
silently collapsed into the log. The M5 appeal endpoint depends on it.

## Setup

```
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env
# add your CEREBRAS_API_KEY to .env

```
